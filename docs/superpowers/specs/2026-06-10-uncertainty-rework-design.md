# Uncertainty.ipynb — переработка в развёрнутый учебный формат

> Дата: 2026-06-10
> Статус: дизайн утверждён, ожидает реализации

## Проблема

Ноутбук `src/notebooks/Uncertainty.ipynb` (32 ячейки, 9 секций) написан в
телеграфном стиле: markdown-обвязка сокращена до криптичных ярлыков
(«кон. вероят. распределение», «комплектная замкн. подкат.»), таблицы с
непрозрачными аббревиатурами, почти нет мотивации «откуда и зачем».
Код при этом развит и снабжён демо. Секция 9 (теория возможностей,
переработана в Фазе 8) заметно полнее остального — она служит эталоном
желаемого объёма.

Цель: привести **весь** ноутбук (введение + 9 секций + сводка) к единому
развёрнутому стилю с тремя каналами восприятия (текст / таблица / визуал)
и явными границами применимости моделей.

## Решение

### Единый скелет каждой секции

Каждая содержательная секция переписывается по фиксированному шаблону:

1. **Мотивация** — какую неопределённость моделируем, откуда возникает задача.
2. **Идея / конструкция** — что за структура, её категорный смысл.
3. **Формализм** — определения, ключевые формулы (делимитеры только `$...$`).
4. **Код + развёрнутая демо** — существующий код сохраняется исполнимым,
   добавляются комментарии «зачем», демо поясняется.
5. **Пример из жизни** — конкретный прикладной кейс.
6. **Границы применимости** — когда модель работает, когда ломается, чем
   отличается от соседних конструкций.
7. **Мостик** — связка со следующей секцией.

### Визуал (каноничный SVG: ASCII-текст, белый фон #ffffff, monospace)

Диаграмма на каждую «тяжёлую» секцию:
- Kleisli-композиция ядер (Гири, секция 3);
- решётка `[0,1]` + t-нормы/t-конормы (Fuzzy, секция 4);
- Kripke-фрейм + отношение достижимости (модальные, секция 5);
- дуальные шкалы `L`/`L̄` (возможности, секция 9 — при необходимости);
- 1–3 дополнительных по ходу (напр. Bayesian network для секции 6,
  граф марковской цепи для секции 7).

Каталог: `src/diagrams/uncertainty/` (создать).

### Обзорная сводка категорных конструкций (3 канала)

В финальной секции «Сводка»:
- **SVG-постер** «категорные конструкции неопределённости»: строки = виды
  неопределённости (Maybe/Either, Dist, Giry, Fuzzy, Modal, Possibility),
  столбцы = категория / эндофунктор-или-обогащение / композиция
  (Kleisli/решётка) / измерение (мера/μ/τ);
- **markdown-таблица** с тем же содержанием (для копируемости и поиска);
- **текстовый абзац**, проговаривающий единую картину.

## Инварианты

- Код везде остаётся исполнимым: **0 ошибок** при прогоне ядра.
- Математические делимитеры — только `$...$` (НЕ `\(...\)`).
- SVG — ASCII-текст, белый фон, без unicode-entity с кириллицей.
- Навигационный футер и связи с `README.ipynb` синхронны (обновить, если
  изменится число ячеек/SVG).
- README остаётся обложкой/навигацией — трогаем только счётчики при
  изменении объёма. Технический план — в `ROADMAP.md` (Фаза 9).

## Стратегия исполнения (два прохода + параллельный визуал)

**Проход 1 — проза.** Весь markdown всех секций переписывается за один
проход (единый голос), + обе сводки-таблицы. Один прогон ядра в конце —
проверка живости кода (0 ошибок).

**Проход 2 — визуал.** SVG-диаграммы независимы и подчинены строгим
правилам стиля → раздаются параллельно агентам (по диаграмме на агента),
собираются в ноутбук, финальный прогон ядра.

Прерывания (лимиты) безопасны: разбивка по секциям/диаграммам
позволяет продолжить с любого места.

## Вне scope

- Изменение Haskell-логики существующего кода (только комментарии и демо).
- Рефакторинг других ноутбуков.
- Переписывание README сверх обновления счётчиков.
