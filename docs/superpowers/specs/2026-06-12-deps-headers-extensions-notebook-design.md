# Дизайн: шапки зависимостей + нулевой ноутбук Extensions

**Дата:** 2026-06-12. **Статус:** утверждено.

## Цель

1. В начале каждого из 24 ноутбуков — автогенерируемая markdown-ячейка со списком
   используемых пакетов, модулей библиотеки курса и расширений GHC с однострочными
   пояснениями.
2. Новый нулевой ноутбук `Extensions.ipynb` — бриф по расширениям Haskell:
   о чём, зачем, в чём соль.

## Решения (из брейншторма)

- **Источник шапки:** автогенерация скриптом + общий словарь пояснений (не вручную).
- **Охват нулевого ноутбука:** расширения курса с примерами + краткий обзор важного
  вне курса (GADTs, TypeFamilies, DataKinds и т.п.).
- **Гранулярность библиотек:** пакеты, не модули (через словарь «префикс модуля → пакет»);
  модули `src/lib` — отдельной строкой.

## Архитектура

### Словарь: `src/scripts/deps/dictionary.json`

Один источник правды. ASCII-ключи, русские значения:

```json
{
  "extensions": {
    "RankNTypes": { "blurb": "типы с forall внутри аргументов", "anchor": "rankntypes" }
  },
  "packages": {
    "Data.Map": "containers", "Data.Map.Strict": "containers",
    "Control.Monad.State": "mtl", "_default": "base"
  },
  "libModules": ["Quantale", "KanExtension", "Bitopos", "Distribution", "SubjectiveModel"]
}
```

Маппинг модуль→пакет — по самому длинному совпадающему префиксу; нет совпадения → `base`.

### Генератор: `src/scripts/deps/gen_headers.js`

Node-скрипт, для каждого `src/notebooks/*.ipynb`:

1. Извлекает `-X`-расширения из setup-ячейки (первая code-ячейка с `:set`).
2. Извлекает все `import`-строки по всем code-ячейкам, сворачивает в множество пакетов;
   импорты модулей из `libModules` — в отдельный список «Библиотека курса».
3. Рендерит markdown-ячейку с фиксированным `id: "deps_header"` и вставляет её сразу
   после титульной markdown-ячейки (первой markdown-ячейки ноутбука); если ячейка
   с этим id уже есть — заменяет source (идемпотентность, без дублей).
4. Если расширение отсутствует в словаре — завершается с ошибкой и списком недостающих
   ключей (защита от молчаливого дрейфа).
5. Запись JSON: `ensure_ascii`-семантика как в курсе — не-ASCII в source допустим,
   формат файла сохраняется (отступ 1, как у Jupyter).
6. Дополнительный режим `--index`: печатает обратный индекс «расширение → ноутбуки»
   (для написания Extensions.ipynb).

`DesignShowcase.ipynb` (0 расширений, 0 импортов) получает шапку только если есть что
показывать; иначе пропускается.

### Формат шапки

```markdown
> **📦 Зависимости**
> **Пакеты:** `base`, `containers`, `mtl`
> **Библиотека курса:** `Quantale`, `SubjectiveModel` (`src/lib`, см. ROADMAP Фаза 15)
> **Расширения:** `RankNTypes` — типы с forall внутри ([→](Extensions.ipynb#rankntypes)) · `FlexibleInstances` — …
```

Строки «Библиотека курса» нет, если ноутбук не грузит `src/lib`.

### Нулевой ноутбук: `src/notebooks/Extensions.ipynb`

Рукописный, стандарт курса: SETUP + TOC + NAV. Структура:

1. **Введение** — что такое language extensions, `:set -X...` vs pragma.
2. **Расширения курса по группам** (каждое — заголовок с HTML-якорем `<a id="...">`,
   мини-пример «без/с расширением», список ноутбуков-потребителей из обратного индекса):
   синтаксис и литералы (OverloadedStrings, LambdaCase, TupleSections),
   классы типов (FlexibleInstances, FlexibleContexts, MultiParamTypeClasses,
   FunctionalDependencies), типы высших рангов (RankNTypes, ScopedTypeVariables),
   deriving (DeriveFunctor, GeneralizedNewtypeDeriving, DeriveGeneric).
   Точный состав групп — по фактическому объединению `-X` из 24 ноутбуков.
3. **За пределами курса** — обзорно: GADTs, TypeFamilies, DataKinds, TypeApplications,
   LinearTypes; по абзацу «в чём соль», код минимальный или отсутствует.
4. **Итоговая таблица** «расширение → ноутбуки курса».

Все примеры исполняются в контейнере gibiansky/ihaskell. Соблюдать критические правила
ROADMAP (`\xNNNN`, без QuantifiedConstraints и т.д.).

## Верификация

1. `gen_headers.js` отрабатывает по всем ноутбукам без ошибок; повторный запуск
   не меняет файлы (идемпотентность) — проверка `git diff --quiet` после второго прогона.
2. JSON-валидность всех 25 ноутбуков node-ом.
3. `nbconvert --execute` для `Extensions.ipynb` в контейнере: 0 ошибок.
4. Существующие ноутбуки не перепрогоняются (вставка markdown не влияет на код);
   выборочно открыть 1–2 в Jupyter и проверить рендер шапки.
